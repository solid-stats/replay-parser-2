  <action>
    Update the storage boundary so artifact writes attempt create-if-absent before falling back to compare/reuse/conflict:

    - Add a small internal outcome enum in `storage.rs` with exact variants `ArtifactPutOutcome::Created`, `ArtifactPutOutcome::AlreadyExists`, and `ArtifactPutOutcome::UnsupportedConditionalWrite`.
    - Add an `ObjectStore` method named `put_artifact_bytes_if_absent` with signature:
      `fn put_artifact_bytes_if_absent<'a>(&'a self, object_key: &'a str, bytes: &'a [u8], content_type: &'a str) -> ObjectStoreFuture<'a, ArtifactPutOutcome>`.
    - The default trait implementation should call the existing unconditional `put_object_bytes` and return `ArtifactPutOutcome::Created`; fake stores can override this to simulate race outcomes.
    - Change `write_artifact_if_absent_or_matching` to:
      1. compute `new_checksum` and `new_size_bytes`;
      2. call `put_artifact_bytes_if_absent(key, bytes, "application/json")`;
      3. return `reused_existing=false` when the outcome is `Created`;
      4. when outcome is `AlreadyExists`, fetch `get_artifact_bytes(key)` and compare exact SHA-256 and byte size;
      5. reuse the existing object only when both checksum and size match;
      6. return `WorkerFailureKind::ArtifactConflict` when existing bytes differ;
      7. when outcome is `UnsupportedConditionalWrite`, preserve the previous get-then-put fallback path but still compare existing bytes if an object is present.
    - Implement `S3ObjectStore::put_artifact_bytes_if_absent` with AWS SDK `put_object().if_none_match("*")`.
    - Map AWS/S3-compatible conditional-exists responses to `ArtifactPutOutcome::AlreadyExists` when service error code/status indicates `PreconditionFailed`, `PreconditionFailedException`, HTTP 412, `ConditionalRequestConflict`, or HTTP 409.
    - Map provider unsupported conditional-write responses to `UnsupportedConditionalWrite` only for low-cardinality service codes such as `NotImplemented`, `NotSupported`, `InvalidRequest`, or HTTP 501/400 responses whose message mentions `If-None-Match`; otherwise preserve retryable `WorkerError::S3`.
    - Keep `WorkerFailureKind::ArtifactConflict` error code exactly `output.artifact_conflict`.
  </action>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Worker instance to S3 artifact key | Multiple workers can compute the same deterministic key and race to write it. |
| RabbitMQ redelivery to durable artifact state | A redelivered job can observe an artifact from a previous delivery attempt. |
| AWS S3 semantics to Timeweb S3-compatible semantics | Conditional-write support may differ by provider. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-07-00-01 | Tampering | S3 artifact write | mitigate | Use conditional create-if-absent when available and exact SHA-256/size comparison on existing keys. |
| T-07-00-02 | Repudiation | Duplicate job result | mitigate | Duplicate redelivery publishes another confirmed completed/failed outcome instead of silently ignoring state. |
| T-07-00-03 | Denial of Service | Provider conditional-write mismatch | mitigate | Unsupported conditional-write responses fall back to compare/reuse/conflict instead of crashing worker startup. |
| T-07-00-04 | Information Disclosure | Storage errors | mitigate | Storage errors report bucket/key/error code only and do not include AWS credentials or AMQP URLs. |
</threat_model>
