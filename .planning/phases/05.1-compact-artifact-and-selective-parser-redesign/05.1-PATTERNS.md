## Existing Interfaces To Reuse

- `parser_core::parse_replay(ParserInput)` remains the pure parser entrypoint for CLI, worker, harness, tests, and benchmarks.
- `ParserInput` keeps bytes, source metadata, parser info, and deterministic options at the boundary.
- `ReplaySource`, `SourceRef`, `SourceRefs`, `RuleId`, `FieldPresence`, and `Diagnostic` remain the provenance/status vocabulary.
- `ReplayMetadata`, `ReplaySideFacts`, `EventActorRef`, `VehicleContext`, `BountyEligibility`, `LegacyCounterEffect`, `AggregateContributionRef`, and `VehicleScoreInputValue` are reusable semantics even when no longer exposed as full `events`/`entities` arrays.
- `parser_contract::schema::parse_artifact_schema()` remains the schema source of truth for the CLI and committed schema.
- `ComparisonReport` categories and impact dimensions remain the detailed evidence vocabulary; Markdown summary should be generated from the same data.

## Existing Test Style

Current tests use public APIs, focused OCAP fixtures, explicit AAA/RITE-style names, schema/example freshness checks, and deterministic serialization assertions. Phase 5.1 should continue this style and strengthen it around:

- absence of top-level compact-contract-forbidden fields;
- compact participant/fact source refs;
- selective parser grep gates;
- benchmark report validator edge cases;
- Markdown comparison output snapshots or string checks.

## Boundary Constraints

Do not add these concerns during Phase 5.1:

- RabbitMQ/S3 worker implementation, ack/nack, or artifact upload behavior;
- PostgreSQL, API, OpenAPI, or public UI implementation;
- replay discovery or production raw fetches;
- canonical player identity matching;
- final bounty calculation or cross-replay vehicle score calculation;
- annual/yearly nomination product support.
