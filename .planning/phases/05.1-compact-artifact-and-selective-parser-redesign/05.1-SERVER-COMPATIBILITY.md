# Phase 5.1 Server Compatibility Review

## Review Scope

This review records the compatibility gate for the Phase 5.1 compact parser
artifact redesign before implementation changes the public parser contract.

Phase 5.1 is a breaking compact contract/schema redesign. The default parser
artifact delivered by future worker integration must become a compact
server-facing result, not a full normalized replay dump.

## Breaking Artifact Changes

removed default top-level fields: entities, events.

Added compact fields planned by Phase 5.1:

- `participants`
- `facts`
- `summaries`

Preserved envelope fields:

- `contract_version`
- `parser`
- `source`
- `status`
- `produced_at`
- `diagnostics`
- `replay`
- `side_facts`
- `failure`
- `extensions`

The old full normalized artifact is not retained as a supported public schema
field set. Deep investigation should use raw replay reprocessing or explicit
future debug/parity tooling rather than ordinary worker ingestion.

## server-2 Ingestion Expectations

`server-2` can persist compact participant refs, combat facts, aggregate
contribution refs, side facts, and projections from the artifact reference it
receives over RabbitMQ/S3 later in Phase 6.

The compact artifact should provide enough observed identity, replay/source
metadata, source references, rule IDs, bounty eligibility/exclusion facts, and
vehicle-score contribution facts for `server-2` to recalculate and audit
statistics without requiring full top-level `entities` or `events` dumps.

Phase 6 still owns the RabbitMQ/S3 worker message shape, artifact key format,
checksum verification, and parse result publication details.

## Boundaries Preserved

- parser does not calculate final bounty points per D-11.
- parser does not calculate final cross-replay vehicle score per D-14.
- parser preserves observed identity only; canonical matching belongs to server-2.
- parser does not write PostgreSQL business tables.
- parser does not own public APIs, moderation workflow, UI, replay discovery,
  raw object staging, or yearly nomination product behavior.
- `replays-fetcher` continues to own external replay discovery, raw S3 object
  writes, source metadata, and staging/outbox records.
- `web` continues to consume `server-2` APIs and generated OpenAPI types rather
  than the parser artifact directly.

## Compatibility Risks

Actual adjacent implementation docs were not present for the parser-ingestion
schema. Only local `gsd-briefs` evidence is available, so the compact contract
implementation required explicit user acceptance before Plan 01 modifies the
public parser artifact contract.

Known risks:

- `server-2` may later need a migration from the older full `events`/`entities`
  expectations to compact `participants`, `facts`, and `summaries`.
- If `server-2` needs raw event rows for PostgreSQL audit tables, it must derive
  them from compact facts or request explicit debug/reprocessing tooling rather
  than depend on ordinary ingestion carrying a full replay-shaped artifact.
- `web` API types may need a later `server-2` OpenAPI update when compact parser
  fields become visible through backend APIs.
