# Phase 5.1: Compact Artifact and Selective Parser Redesign - Context

**Gathered:** 2026-04-29T13:37:29+07:00
**Status:** Ready for planning

<domain>
## Phase Boundary

Phase 5.1 corrects the Phase 5 UAT rejection before RabbitMQ/S3 worker work
starts. It redesigns the ordinary server-facing parser artifact as a compact
statistics/facts result, removes full normalized event/entity dumps from the
default ingestion path, replaces the current full JSON-to-JSON parser hot path
with a selective compact path, and makes benchmark/comparison evidence
reviewable against selected and whole-list/corpus workloads.

This phase may revise Phase 2-5 contract and harness decisions, but it must keep
canonical player matching, PostgreSQL persistence, public APIs, web UI behavior,
replay discovery, production raw replay fetching, and annual/yearly nomination
product support outside `replay-parser-2`.

</domain>

<decisions>
## Implementation Decisions

### Artifact Cutline
- **D-01:** Treat the compact artifact redesign as a breaking contract/schema
  redesign. Downstream planning must update contract/schema versioning, schema
  export, examples, fixtures, comparison expectations, and README commands.
- **D-02:** The default artifact is a compact server-facing stats/facts
  artifact. It contains replay/source metadata, minimal observed participant
  references, compact stat/combat facts, per-replay summaries, side facts,
  diagnostics, parser/contract version data, and structured failure/status data.
- **D-03:** Remove the current top-level full `entities` and `events` dumps from
  the default artifact. Do not preserve them as supported public schema fields.
- **D-04:** Exact compact section names and Rust type names are planner
  discretion; this context locks semantics, not final field spelling.
- **D-05:** `replay-parser-2 parse` writes the same compact default artifact that
  the future worker will deliver to `server-2`. Local CLI output must not default
  to a fuller artifact than worker ingestion.
- **D-06:** Compact participant references must include enough observed identity
  for ordinary `server-2` ingestion and canonical matching: source entity ID,
  observed name, side, group/squad/role where available, SteamID
  presence/value where available, and source references. Do not include full
  entity snapshots.
- **D-07:** The old full normalized artifact is not a supported public output or
  public schema after this redesign. Historical evidence can remain in git
  history/generated reports; detailed investigation should use raw replay
  reprocessing or explicit tooling, not ordinary ingestion.
- **D-08:** Success, partial, failed, and skipped outputs should share the same
  compact envelope. Failed/skipped artifacts fill source/version/status/failure
  details and omit unavailable stat sections.
- **D-09:** Because the artifact shape changes `server-2` ingestion, planning
  must include `server-2` compatibility review or explicit user acceptance
  before contract implementation starts.
- **D-10:** Use a relative compactness target in planning: the default artifact
  must be materially smaller than raw OCAP/full artifact output. The exact size
  threshold must be set from benchmark/corpus evidence.

### Audit Evidence and Server Calculation Boundary
- **D-11:** Parser must not calculate final bounty points. Final bounty depends
  on previous rotation and cross-replay player/squad effectiveness, so it is
  owned by `server-2`.
- **D-12:** Parser emits compact per-replay combat/stat facts needed by
  `server-2`, such as valid enemy kill facts, killer/victim observed identity,
  side context, frame/source refs, and exclusion reasons for teamkill, suicide,
  null killer, or unknown actor cases.
- **D-13:** Default artifact contains compact facts plus per-replay summaries.
  Summaries support parity/review and fast ingestion sanity checks; facts remain
  the source for server recalculation. No cross-replay/global/rotation totals
  are parser-owned.
- **D-14:** Vehicle score follows the same boundary: parser emits issue #13
  vehicle-kill/teamkill weighted contribution facts and per-replay denominator
  eligibility; `server-2` or the parity harness computes final cross-replay
  vehicle score.
- **D-15:** Raw replay reprocessing is the authoritative fallback for deep
  investigation. Optional debug/detail tooling may exist later, but it must not
  be required for ordinary worker ingestion.

### Selective Parser Path
- **D-16:** Build a compact-first parser path. The normal/default path must no
  longer parse, clone, and serialize the whole replay through a full
  `serde_json::Value` artifact.
- **D-17:** No full `serde_json::Value` DOM is allowed on the normal ingestion
  path. DOM-based helpers may be used in tests or explicit debug tooling only.
- **D-18:** Selective extraction must preserve raw source coordinates directly
  while reading relevant tuples/fields: event index, frame, entity ID, JSON path,
  and rule ID.
- **D-19:** User explicitly allowed a full parser-core rewrite override despite
  risk. Planning must record this as high risk: a rewrite can replace current
  core semantics, but Phase 3/4 edge-case behavior, old-parser parity, source
  reference invariants, coverage, and acceptance tests must prove the new path
  before Phase 6 proceeds.
- **D-20:** Minimum acceptance is implemented and measured, not merely planned:
  compact normal path exists, normal-path full DOM is removed, quality gates
  pass, and benchmark reports include selected and whole-list/corpus evidence.

### Benchmark and Comparison Gates
- **D-21:** Benchmark evidence before Phase 6 must include both selected
  replay/sample checks and whole-list/corpus evidence, or an explicit rationale
  that whole-list/corpus evidence was unavailable. One-replay timing alone
  cannot pass Phase 5.1.
- **D-22:** The roughly 10x target remains blocking unless the report explains
  bottleneck and parity status and the user explicitly accepts the gap.
- **D-23:** Benchmark reports must include raw input bytes, compact artifact
  bytes, artifact/raw ratio, selected/full workload identity, parse-only,
  aggregate/facts-only where meaningful, end-to-end throughput, memory/RSS where
  practical, parity status, and 10x status.
- **D-24:** Comparison output should be human-first. Default comparison review
  should be Markdown summary with counts, top actionable diffs, and next-action
  notes. Structured JSON evidence/details may be generated for audit/debug, but
  the user does not require comparison reports to be CI blockers by default.
- **D-25:** Phase 5.1 acceptance must not leave unexplained parity blockers.
  `human_review` is acceptable only when the report is readable, top diffs are
  categorized, and the next action or explicit acceptance is recorded.

### the agent's Discretion
- Exact section names, Rust module split, schema type names, and file names are
  planner discretion if they preserve the decisions above.
- Exact artifact-size threshold is planner discretion after corpus/benchmark
  evidence, but the result must prove material compactness.
- Exact selective parsing technique is planner discretion; candidates include
  typed Serde models, borrowed/raw-value slices, or a purpose-built top-level
  extractor, provided the normal path avoids full replay DOM translation.
- Exact Markdown/JSON comparison file layout is planner discretion if default
  review is summary-first and optional machine data does not become a required
  CI gate without a later decision.

</decisions>

## Existing Code Insights

### Reusable Assets
- `parser_core::parse_replay(ParserInput)` is the existing pure parser entry
  point and should remain the adapter boundary even if its internals are
  rewritten.
- `ParserInput` already separates replay bytes, source metadata, parser info,
  and deterministic options.
- Current raw accessors in `crates/parser-core/src/raw.rs` already know how to
  observe top-level fields, connected events, killed events, and source JSON
  paths.
- Existing Phase 3/4 tests and fixtures cover metadata, entity drift,
  connected-player backfill, combat events, aggregate projection, side facts,
  vehicle score, and deterministic output. They are the practical safety net for
  any rewrite.
- `parser-harness` already owns comparison and benchmark report vocabulary; it
  is the right place for summary-first comparison and compact artifact-size
  evidence.

### Established Patterns
- Parser-core is pure and transport-free. File I/O, CLI formatting, old-parser
  invocation, generated reports, benchmarking, timestamps, RabbitMQ, S3, and
  database concerns belong in adapters/harnesses.
- Contract/schema tests generate schema from Rust types and compare against
  committed schema/examples.
- Deterministic output uses stable ordering and avoids wall-clock values in
  parser-core output.
- Existing comparison reports carry Phase 1 mismatch categories and
  parser/server/web impact dimensions.
- Bulky generated evidence belongs under ignored `.planning/generated/` paths,
  not committed fixture directories.

### Integration Points
- Replace `ParseArtifact` default fields so full `entities`/`events` are no
  longer part of ordinary success/partial artifacts.
- Rewrite `crates/parser-core/src/artifact.rs` so normal parsing no longer
  starts with `serde_json::from_slice::<Value>` over the entire replay.
- Update CLI parse/schema behavior to emit compact artifacts and schema.
- Update benchmark report generation/validation to include compact artifact
  size and whole-list/corpus evidence.
- Update comparison harness output so the default review surface is Markdown
  summary-first, with optional structured detail artifacts.
- Add explicit `server-2` compatibility review before implementation because
  this changes the artifact shape consumed by worker integration.

<specifics>
## Specific Ideas

- The user explicitly challenged any implication that parser calculates bounty.
  The locked boundary is: parser emits compact kill/stat facts; `server-2`
  calculates final bounty points using previous rotation and cross-replay state.
- Vehicle score is also not finalized by parser across replays. Parser emits
  issue #13 contribution facts and denominator eligibility; server/parity
  harness computes the final cross-replay score.
- The user accepted a high-risk full rewrite override for parser-core rather
  than limiting the change to a small hot-path refactor.
- The user does not require comparison report evidence to be CI-blocking by
  default. Human-readable Markdown summary is the preferred default review
  surface; optional structured evidence can still be generated.
- Raw replay reprocessing is acceptable as the fallback for deep detail after
  removing the full normalized artifact from default output.

</specifics>
