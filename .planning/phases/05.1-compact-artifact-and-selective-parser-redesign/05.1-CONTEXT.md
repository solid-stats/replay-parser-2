# Phase 5.1 Context: Compact Artifact and Selective Parser Redesign

**Status:** Inserted urgent phase, not planned yet
**Inserted:** 2026-04-29
**Inserted after:** Phase 5, `CLI, Golden Parity, Benchmarks, and Coverage Gates`

## Why This Exists

Phase 5 made the CLI, comparison harness, coverage gate, fault report gate, and benchmark report executable, but UAT rejected the parser direction:

- The default parser output largely turns one 10-15 MB OCAP JSON file into another large JSON artifact.
- The measured speedup is far below the roughly 10x target.
- The comparison report is too large for practical human review.

This phase must correct the product direction before Phase 6 worker integration. Worker mode should not be built around an oversized artifact contract or a parser hot path that performs a full JSON-to-JSON translation.

## User Decisions Captured

- Default server output should be compact.
- Full normalized event/entity dumps should be removed from ordinary server ingestion.
- Heavy debug/parity evidence can be optional, but should not slow the normal parser path.
- Annual/yearly nomination statistics should not force a large v1 side artifact; future yearly work can reprocess raw OCAP files when needed.
- Performance work should focus on selective parsing rather than only optimizing the current full-artifact core.

## Planning Targets

Phase planning should define:

1. The compact server-facing artifact shape.
2. The exact detail that remains in default contribution/source evidence.
3. What, if anything, becomes optional sidecar/debug output.
4. The selective parser strategy and measurable acceptance criteria.
5. Artifact-size reporting and benchmark pass/fail rules.
6. A summary-first comparison report format with detailed evidence separated from the default review surface.
7. Compatibility checks needed for `server-2` before Phase 6.
