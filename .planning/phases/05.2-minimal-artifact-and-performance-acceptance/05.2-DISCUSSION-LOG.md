## Preloaded User Intent

The user reported that the parsed artifact is still too large and should contain
only minimal information:

- players;
- player kills, deaths, teamkills, and similar counters;
- who killed whom and what weapon/vehicle was used;
- which vehicles/static weapons were destroyed.

Detailed data may exist only behind a separate debug flag because problematic
replays can be regenerated quickly. The user also decided to remove GitHub issue
#13 vehicle score from v1 and later requested performance targets of x3 on one
replay and x10 on the full corpus.

## Questions and Answers

### Issue #13 Scope

**Question:** How exactly should GitHub issue #13 vehicle score be removed from
v1?

**Answer:** Remove code. Remove vehicle score types/modules/projections/tests
from v1, leaving ordinary `vehicleKills`, `killsFromVehicle`, and
`destroyed_vehicles`.

### Debug Sidecar Depth

**Question:** What should the explicit debug sidecar write in Phase 5.2?

**Answer:** Full normalized detail.

### Selected Replay Policy

**Question:** How should the one-replay x3 gate choose its replay?

**Answer:** Automatic large replay selection by a stable rule, with selected
identity recorded in the benchmark report.

### Players Scope

**Question:** Who belongs in minimal artifact `players[]`?

**Answer:** Observed players only, including connected/backfilled and
legacy-eligible participants. Do not include every entity or stats-only rows.

### Kills Scope

**Question:** Which events belong in default `kills[]`?

**Answer:** All player deaths: enemy kills, teamkills, suicides, null-killer
deaths, and unknown player deaths.

### Default Coordinates

**Question:** Which source coordinates can remain in default rows if source refs
move to debug?

**Answer:** No coordinates in default rows.

### Player Stats Fields

**Question:** Should `player_stats[]` store only counters or also legacy-derived
fields?

**Answer:** Counters only. Derived ratios and scores can be recalculated by
`server-2`.

### Destroyed Vehicles Scope

**Question:** Which vehicle/static events belong in `destroyed_vehicles[]`?

**Answer:** All vehicle/static destructions by players, including
enemy/friendly/unknown-side cases, with weapon and attacker vehicle context and
without issue #13 weights.

### Full-Corpus Benchmark Fairness

**Question:** How should the x10 full-corpus benchmark compare against the old
parser?

**Answer:** End-to-end sequential: old parser with `WORKER_COUNT=1`, new parser
sequentially writing default artifacts.

### Coordinate Confirmation

**Question:** Does "no coordinates" mean no `frame/time` either in default
`kills[]`/`destroyed_vehicles[]`?

**Answer:** Yes. No frame/time in default; all coordinates go to debug.

### Debug Contract Status

**Question:** Is full normalized debug sidecar a public contract?

**Answer:** Internal only. It should be deterministic and useful for
investigation, but not server-facing contract.

### Corpus Failure Policy

**Question:** What should full-corpus gate do with raw files that do not parse?

**Answer:** Zero failures. Any failed/skipped artifact blocks the gate unless a
later explicit allowlist decision is made.

### Minimal Parity Gate

**Question:** Is old-vs-new parity still required after moving to minimal
artifact?

**Answer:** Yes, through a derived legacy view from minimal tables, not by
matching old JSON shape directly.

### Default JSON Format

**Question:** How should ordinary `parse` serialize the default artifact?

**Answer:** Minified by default. Pretty output can be explicit.

## Resulting Planning Guidance

Phase 5.2 should plan a breaking minimal contract redesign, not another compact
artifact tweak. The highest-risk areas are removing issue #13 implementation,
preventing debug detail from touching the default performance path, and building
credible end-to-end sequential benchmark evidence over all raw files with zero
failures.
