# Phase 5.2: Minimal Artifact and Performance Acceptance - Context

**Gathered:** 2026-05-01T08:16:10+07:00
**Status:** Ready for planning

<domain>
## Phase Boundary

Phase 5.2 replaces the Phase 5.1 compact artifact with a minimal flat v1
statistics artifact, removes GitHub issue #13 vehicle score from v1, adds an
explicit debug sidecar for detailed investigation, and proves performance and
artifact-size acceptance before RabbitMQ/S3 worker integration.

This phase may revise the Phase 2-5.1 public parser contract, schema,
examples, CLI parse behavior, harness reports, comparison surfaces, and tests.
It must keep replay discovery, raw object staging, PostgreSQL persistence,
canonical player matching, public APIs, web UI behavior, and final bounty
points outside `replay-parser-2`.
</domain>

<decisions>
## Locked Decisions

### Default Minimal Artifact

- **D-01:** The default v1 success artifact uses minimal flat tables:
  `players[]`, `player_stats[]`, `kills[]`, `destroyed_vehicles[]`, and
  `diagnostics[]`, plus only the minimal contract/status/source/replay metadata
  needed for deterministic ingestion.
- **D-02:** `players[]` includes observed player participants only, including
  connected/backfilled and legacy-eligible players. It does not include every
  OCAP entity; vehicles/static weapons appear only as kill/destruction context.
- **D-03:** `player_stats[]` contains counters only. Keep kills, deaths,
  teamkills, suicides/null deaths where modeled, `vehicleKills`,
  `killsFromVehicle`, and similar replay-local counts. Do not put derived
  `kdRatio`, `score`, `killsFromVehicleCoef`, or final bounty/rotation values
  in the default artifact.
- **D-04:** `kills[]` includes all player death rows needed to reconstruct
  current counters and bounty inputs: enemy kills, teamkills, suicides,
  explicit null-killer deaths, and unknown player deaths.
- **D-05:** Default `kills[]` and `destroyed_vehicles[]` rows do not include
  source coordinates. No frame, time, event index, JSON path, rule ID, source
  ref dump, normalized event snapshot, or entity snapshot belongs in default
  rows. These details are debug-sidecar-only.
- **D-06:** Default kill/destruction rows still include compact identity and
  gameplay context: observed killer/victim player references where available,
  enemy/teamkill/suicide/null/unknown classification, weapon, attacker vehicle
  context, destroyed entity type/name/class where available, and enough
  side/context data for `server-2` to derive current stats and bounty inputs.
- **D-07:** `destroyed_vehicles[]` includes all vehicle/static destructions by
  players: enemy, friendly, and unknown-side cases. It preserves weapon and
  attacker vehicle context but no issue #13 weights.
- **D-08:** Default artifact JSON is minified for ordinary `parse` output.
  Human-readable pretty output can be an explicit option, not the default.

### Issue #13 Removal

- **D-09:** GitHub issue #13 vehicle score is removed from v1 implementation,
  not merely hidden. Planning should remove `vehicle_score` contract types,
  projections, tests, schema/example fields, docs, and acceptance criteria from
  the v1 path.
- **D-10:** V1 keeps ordinary `vehicleKills`, `killsFromVehicle`, weapon,
  attacker vehicle, and destroyed-vehicle facts. If vehicle score returns later,
  it can be rebuilt from raw replay reprocessing or a future explicit feature.

### Debug Sidecar

- **D-11:** Detailed investigation output is written only behind an explicit
  debug flag such as `--debug-artifact <path>`.
- **D-12:** The debug sidecar should be full normalized detail, including
  normalized entities/events/source refs/rule IDs, sufficient for local
  investigation and parity debugging.
- **D-13:** The debug sidecar is internal tooling, not the server-facing public
  contract. It should be deterministic and tested, but it does not get the same
  compatibility guarantees as the default parse artifact.
- **D-14:** The normal/default path must not build full normalized debug detail
  unless the debug flag is used.

### Performance and Size Gates

- **D-15:** Benchmark evidence must first capture the current old/new baseline
  on the chosen workloads, then prove the Phase 5.2 implementation.
- **D-16:** The selected single-replay x3 gate uses an automatic stable
  large-replay selection policy. The script should choose a large existing raw
  replay by a deterministic rule and record the selected replay identity in the
  report.
- **D-17:** The full-corpus x10 gate is end-to-end sequential evidence:
  old parser uses deterministic `WORKER_COUNT=1`, and new parser processes raw
  files sequentially while writing default artifacts.
- **D-18:** The full-corpus gate covers all files in `~/sg_stats/raw_replays`,
  not just replay-list rows. Every raw file must be attempted.
- **D-19:** The full-corpus gate has a zero-failure policy. Failed/skipped files
  block the gate unless planning creates an explicit user-approved allowlist;
  the report must still include success/failure/skip counts and triage.
- **D-20:** Successful all-raw artifacts must pass the size gate: median
  artifact/raw ratio <= 5%, p95 artifact/raw ratio <= 10%, and every successful
  default parse artifact <= 100 KB (100,000 bytes). Tiny fixtures can be
  reported separately but do not define acceptance.
- **D-21:** Old-vs-new parity should compare a derived legacy view from minimal
  tables against old results. The default minimal JSON shape itself does not
  need to match the old parser output shape.

### Cross-App Boundary

- **D-22:** Product-owner compatibility acceptance must be recorded before or
  during implementation: `server-2` will adapt later to the minimal flat
  artifact. Parser still does not own canonical identity, PostgreSQL business
  persistence, public APIs, web presentation, replay discovery, or bounty payout
  rules.

### Post-Execution Acceptance Update

- **D-23:** Product-owner acceptance on 2026-05-02 supersedes the original
  x3/x10 blocking policy. Benchmark reports must still record selected and
  all-raw old/new timings plus historical x3/x10 statuses, but the current
  measured performance is accepted for Phase 6 readiness.
- **D-24:** Product-owner acceptance on 2026-05-02 supersedes the original p95
  artifact/raw ratio blocker. Median and p95 ratios remain reported trend
  evidence; the active all-raw size gate is hard max artifact bytes
  `<= 100000` and `oversized_artifact_count == 0`.
- **D-25:** Product-owner acceptance on 2026-05-02 accepts the 4 known
  malformed/non-JSON all-raw files when the cached old parser baseline reports
  the same failure count and the new parser failure paths match the committed
  accepted-failures evidence.
</decisions>

<code_context>
## Current Code Insights

- Current selected generated evidence shows why Phase 5.2 is needed:
  `new-selected-artifact.json` is about 2.1 MiB while the old selected artifact
  is about 60 KiB. The raw selected replay
  `/home/afgan0r/sg_stats/raw_replays/2026_04_19__21_50_28__1_ocap.json` is
  2,721,844 bytes.
- In that selected artifact, `participants` has 143 rows, `facts.combat` has
  64 rows, and `facts.aggregate_contributions` has 332 rows. Compact sections
  still carry large repeated `FieldPresence` source payloads:
  `participants` about 358 KiB compacted, `facts.combat` about 410 KiB,
  aggregate contributions about 269 KiB, and summaries about 244 KiB.
- `ParseArtifact` no longer has top-level full `entities`/`events`, but the
  compact fields are still audit-heavy for the new default.
- `CombatFact` includes frame, event index, killer/victim `FieldPresence`,
  `vehicle_context`, bounty state, legacy counter effects, source refs, and
  rule ID. Phase 5.2 default rows must be much flatter and omit coordinates and
  source refs.
- `ParseFactSection.aggregate_contributions` duplicates much of the data needed
  for projections. Phase 5.2 should derive minimal `player_stats`, `kills`, and
  `destroyed_vehicles` directly or through internal accumulators, then keep
  detailed contribution refs only in debug output if needed.
- `derive_aggregate_section` currently emits projection keys:
  `legacy.player_game_results`, `legacy.relationships`,
  `legacy.game_type_compatibility`, `legacy.squad_inputs`,
  `legacy.rotation_inputs`, `bounty.inputs`, `vehicle_score.inputs`, and
  `vehicle_score.denominator_inputs`. Phase 5.2 should replace this default
  shape with flat minimal tables and remove `vehicle_score.*`.
- `crates/parser-core/src/events.rs` has the combat semantics that should be
  preserved: enemy kill, teamkill, suicide, null-killer death,
  vehicle destroyed, and unknown.
- `scripts/benchmark-phase5.sh` currently uses a tiny CI fixture for selected
  evidence, an optional curated replay, and optional `RUN_PHASE5_FULL_CORPUS`.
  The full-corpus loop currently reads `replaysList.json`, skips missing files,
  writes pretty artifacts, and fails on first CLI failure. Phase 5.2 needs a
  stronger all-raw runner over `~/sg_stats/raw_replays` with zero failed/skipped
  artifacts and complete counts.
- The CLI currently serializes parse output with `serde_json::to_vec_pretty`.
  Phase 5.2 default output should use minified JSON; pretty output should be
  explicit.
</code_context>

<risks>
## Planning Risks

- Removing issue #13 is a real code deletion/refactor, not a documentation-only
  change. Expect contract, schema, examples, aggregate tests, vehicle-score
  tests, docs, and planning assertions to move together.
- The zero-failure all-raw gate may expose malformed or unsupported historical
  files. If any exist, the plan must either fix parser tolerance or ask for an
  explicit allowlist decision; silent skips cannot pass.
- Removing all default coordinates, including frame/time, means bounty/debug
  audit that needs temporal evidence must use the debug sidecar or raw replay
  reprocessing. This is an intentional user decision for Phase 5.2.
- Full normalized debug output must not contaminate default performance. The
  planner should keep debug construction behind the explicit flag and test that
  ordinary parse does not serialize/build debug payloads.
- Minimal default artifact is a breaking contract change after Phase 5.1. Update
  contract versioning, schema, examples, comparison surfaces, benchmark
  validators, README, and server compatibility notes together.
</risks>
