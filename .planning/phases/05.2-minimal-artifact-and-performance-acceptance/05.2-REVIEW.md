# Phase 05.2: Code Review Report

**Reviewed:** 2026-05-01T04:15:25Z
**Depth:** deep
**Files Reviewed:** 41
**Status:** issues_found

## Summary

Deep review found three blockers in the Phase 05.2 acceptance surface. The benchmark checker still exits successfully for reports whose acceptance gates are unknown or failed, partial default artifacts still serialize debug-only provenance through diagnostics, and the committed schema allows removed/debug fields as undeclared extras. I also found script filename-safety and debug-sidecar correctness warnings.

## Critical Issues

### CR-01: [BLOCKER] Benchmark Report Checker Accepts Non-Acceptance Reports

**File:** `crates/parser-harness/src/bin/benchmark-report-check.rs:25`

**Issue:** `benchmark-report-check` only calls `report.validate()` and then prints `benchmark_report_valid=true`. `BenchmarkReport::validate()` allows failed/unknown gates when triage text is present (`crates/parser-harness/src/benchmark_report.rs:147` and `:247`). The current generated report has selected `x3_status=unknown`, selected `artifact_size_status=fail`, selected `parity_status=not_run`, and all-raw statuses `unknown`, yet the checker exits 0 and prints `benchmark_report_valid=true`. Because `scripts/benchmark-phase5.sh --ci` ends by invoking this checker, a structurally valid smoke report can pass the command used in the broader gate without satisfying Phase 5.2 acceptance.

**Fix:**
```rust
impl BenchmarkReport {
    pub fn validate_acceptance(&self) -> Result<(), BenchmarkReportValidationError> {
        self.validate()?;
        if self.selected_large_replay.x3_status != GateStatus::Pass
            || self.selected_large_replay.parity_status != ParityStatus::Passed
            || self.selected_large_replay.artifact_size_status != GateStatus::Pass
            || self.all_raw_corpus.x10_status != GateStatus::Pass
            || self.all_raw_corpus.size_gate_status != GateStatus::Pass
            || self.all_raw_corpus.zero_failure_status != GateStatus::Pass
        {
            return Err(BenchmarkReportValidationError::AcceptanceGatesNotPassed);
        }
        Ok(())
    }
}
```
Call that from the CI checker, or split the CLI into explicit `--structural` and `--acceptance` modes and make `scripts/benchmark-phase5.sh --ci` use acceptance mode.

### CR-02: [BLOCKER] Partial Default Artifacts Still Leak Debug-Only Provenance

**File:** `crates/parser-cli/src/main.rs:273`

**Issue:** `public_parse_artifact()` strips source refs only from `replay` metadata. It leaves `diagnostics` untouched, while parser-core places diagnostics into the default artifact (`crates/parser-core/src/artifact.rs:67`). Any partial parse with schema drift therefore writes `diagnostics[].source_refs[]` with `rule_id`, `json_path`, `frame`, and `event_index` in the ordinary default artifact. This violates the Phase 5.2 rule that source refs, rule IDs, frame/time/event indexes, and detailed provenance are debug-sidecar-only. I verified this with `metadata-drift.ocap.json`; the default CLI artifact contains `diagnostics[0].source_refs[0].rule_id`.

**Fix:** Change the default contract to use a minimal diagnostic row without source refs, and keep full diagnostics in `DebugParseArtifact`. For example:
```rust
pub struct MinimalDiagnosticRow {
    pub code: String,
    pub severity: DiagnosticSeverity,
    pub message: String,
    pub parser_action: String,
}
```
Then map full diagnostics to this row before constructing or serializing the default `ParseArtifact`, and add tests that parse a partial fixture and assert no recursive `source_refs`, `rule_id`, `json_path`, `frame`, or `event_index` keys exist in the default CLI output.

### CR-03: [BLOCKER] Schema Validation Still Allows Removed and Debug-Only Fields

**File:** `crates/parser-contract/src/schema.rs:15`

**Issue:** The generated root schema lists minimal properties, but it never closes the object against unknown top-level fields. In JSON Schema, missing `additionalProperties`/`unevaluatedProperties` means extras are allowed. As a result, an artifact can include removed/default-forbidden fields such as `vehicle_score`, `events`, `entities`, or `source_refs` and still validate against `schemas/parse-artifact-v3.schema.json`. That undercuts the Phase 5.2 contract removal and makes schema validation unable to catch contamination of the default artifact.

**Fix:**
```rust
fn close_default_artifact_schema(schema: &mut Schema) {
    schema
        .as_object_mut()
        .expect("root schema should be an object")
        .insert("unevaluatedProperties".to_owned(), json!(false));
}

pub fn parse_artifact_schema() -> Schema {
    let mut schema = schemars::schema_for!(ParseArtifact);
    enforce_status_failure_invariants(&mut schema);
    enforce_source_ref_evidence_invariants(&mut schema);
    add_aggregate_value_helper_definitions(&mut schema);
    close_default_artifact_schema(&mut schema);
    schema
}
```
Also add schema tests that reject a success example with top-level `vehicle_score`, `entities`, `events`, and `source_refs` fields. Apply the same closure to minimal row definitions if downstream validation must reject row-level debug fields too.

## Warnings

### WR-01: [WARNING] Benchmark Script Breaks on Paths and Metadata with Spaces

**File:** `scripts/benchmark-phase5.sh:89`

**Issue:** The script extracts JSON fields with whitespace-delimited `read` from Python `print(...)` output at lines `89` and `147`. A selected replay path containing spaces corrupts `selected_path`/`selected_raw_bytes`, and mission names with spaces corrupt `selected_mission`/`selected_game_type`. That can make the old selected baseline unavailable or run against wrong metadata even when the corpus is present.

**Fix:** Pass structured data instead of space-delimited fields. One shell-compatible option:
```bash
mapfile -d '' -t selected_fields < <(
  python3 - "$SELECTED_INFO" <<'PY'
import json, sys
info = json.load(open(sys.argv[1], encoding="utf-8"))
for key in ("path", "raw_bytes", "sha256", "source"):
    sys.stdout.write(str(info[key]) + "\0")
PY
)
selected_path=${selected_fields[0]}
selected_raw_bytes=${selected_fields[1]}
```
Use the same null-delimited or JSON-file approach for replay-list metadata.

### WR-02: [WARNING] Debug Sidecar Can Overwrite the Default Artifact

**File:** `crates/parser-cli/src/main.rs:206`

**Issue:** `parse_command()` writes the default artifact and then writes the debug artifact to `--debug-artifact` without rejecting path conflicts. If `--output` and `--debug-artifact` are the same path, the command succeeds but the ordinary output file is replaced by the full debug artifact containing `entities`, `events`, `source_refs`, `rule_id`, `frame`, and `event_index`. That defeats the CLI’s default-minimal guarantee for a simple argument mistake.

**Fix:** Reject identical paths before writing either output.
```rust
if debug_artifact.is_some_and(|path| path == output) {
    return Err(CliError::DebugArtifactConflictsWithOutput);
}
```
Add a CLI test that passes the same path for both flags and asserts the command fails without writing debug content to the default artifact path.

### WR-03: [WARNING] Debug Event Source Refs Use the Wrong Rule IDs

**File:** `crates/parser-core/src/events.rs:321`

**Issue:** `build_teamkill_event()` constructs the event source ref with `VEHICLE_DESTROYED_RULE_ID`, then sets the event `rule_id` to `TEAMKILL_RULE_ID`. `build_friendly_vehicle_destroyed_event()` has the inverse mismatch at line `438`: the source ref is built with `TEAMKILL_RULE_ID`, while the event `rule_id` is `VEHICLE_DESTROYED_RULE_ID`. Debug sidecars therefore contain contradictory provenance for exactly the audit fields Phase 5.2 moved behind `--debug-artifact`.

**Fix:**
```rust
// Player teamkill
let source_ref = event_source_ref(context, observation, TEAMKILL_RULE_ID);

// Friendly vehicle destruction: keep source_ref and event rule consistent.
let source_ref = event_source_ref(context, observation, VEHICLE_DESTROYED_RULE_ID);
```
Add debug-sidecar assertions that every normalized event’s top-level `rule_id` matches the rule carried by its event source refs.
